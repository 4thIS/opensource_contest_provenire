## 1. 전체 그림 — 무엇이 남았나

```
[내 PR의 코드]                        [카피레프트 코퍼스]
      │                                      │
      │  ① 정규화 + 지문      ✅ 완료         │  ② 코퍼스 수집 + 지문 DB   ← WP-A
      ▼                                      ▼
   set[int] ─────────────────────────►  index.search()               ← WP-A
                                             │
                                             ▼
                                        후보 Hit 목록
                                             │
                              ③ 정밀 대조 + 오탐 제거                 ← WP-C
                                             │
                                             ▼
                              ④ PR 인라인 코멘트                      ← WP-B
```

## 2. 작업 패키지 (WP)

각 WP는 **독립적으로 개발 가능**하다. 담당자를 나눠 붙이면 된다.

---

### 🅐 WP-A — 코퍼스 & 인덱스 *(가장 큰 덩어리)*

> **"무엇과 비교할 것인가"** — 카피레프트 코드의 지문 DB를 만든다.

| ID | 작업 | 산출물 |
|---|---|---|
| A-1 | **코퍼스 선정·수집** — GitHub API로 인기 GPL/AGPL/LGPL 프로젝트 Top-N 선정, 소스 다운로드 | `index/corpus.py` |
| A-2 | **청킹 + 지문 생성** — 함수 단위로 쪼개고 지문을 뜬다 | `index/builder.py` |
| A-3 | **지문 DB 저장** — SQLite 또는 parquet | `index/store.py` |
| A-4 | **LSH 검색** — MinHash LSH로 후보를 빠르게 찾는다 (`datasketch`) | `index/search.py` |

**완료 조건 (DoD)**
- [ ] 카피레프트 프로젝트 **최소 30개** 인덱싱 완료
- [ ] `index.search(fp)` 가 **1초 이내**에 후보를 반환
- [ ] 인덱스에 **원본 코드를 저장하지 않는다** (지문 해시만) ← 저작권 안전
- [ ] 라이선스 메타(SPDX ID, URL) 함께 저장

**주의**
- ⚠️ **GPL 코드를 저장소에 커밋하지 않는다.** 캐시는 `.gitignore` 처리.
- 인덱스 파일 자체는 배포 가능(지문 해시일 뿐이므로).

**난이도**: 중상 · **의존성**: 없음 (지금 바로 시작 가능)

### 🅑 WP-B — 스캐너 & PR 게이트 *(제품의 얼굴)*

> **"어디서 실행할 것인가"** — 개발자의 PR에 붙인다.

| ID | 작업 | 산출물 |
|---|---|---|
| B-1 | **`provenire scan`** — 파일/디렉토리를 인덱스와 대조 | `cli.py` 확장 |
| B-2 | **git diff 모드** — `--diff HEAD~1`, PR에서 **추가된 줄만** 검사 (빠름) | `adapters/git.py` |
| B-3 | **GitHub Action** — `action.yml`, PR 인라인 코멘트 | `.github/action/` |
| B-4 | **리포트 포맷** — 콘솔 / SARIF / PR 코멘트 | `report/` |

**완료 조건 (DoD)**
- [ ] `provenire scan ./src --against copyleft` 동작
- [ ] PR을 열면 **자동으로 코멘트**가 달린다 (원본 링크 · 라이선스 · 유사도)
- [ ] `fail-on: critical` 설정 시 **PR 체크 실패**
- [ ] PR diff 검사가 **수 초 내** 완료

**난이도**: 중 · **의존성**: WP-A의 `index.search()` (모킹으로 먼저 개발 가능)

### 🅒 WP-C — 정확도 *(오탐과의 싸움 — 품질의 핵심)*

## 3. 인터페이스 계약 ⭐ *(이것만 합의하면 서로 안 막힌다)*

두 사람이 **각자 독립적으로** 개발하려면 이 시그니처를 먼저 고정해야 한다.

```python
# ─────────── WP-A 담당자가 만드는 것 ───────────
from dataclasses import dataclass

@dataclass(frozen=True)
class Hit:
    project: str        # "qutebrowser"
    file: str           # "qutebrowser/utils/utils.py"
    symbol: str | None  # "elide_filename"  (함수 단위 매칭 시)
    license: str        # "GPL-3.0-or-later"  (SPDX)
    url: str            # 원본 코드 링크
    shared: int         # 공유 지문 개수
    total: int          # 원본 청크의 지문 개수

class Index:
    def search(self, fp: set[int], top_k: int = 10) -> list[Hit]:
        """지문을 받아 유사한 카피레프트 코드 후보를 반환한다."""


# ─────────── WP-B/C 담당자가 쓰는 것 ───────────
from provenire import Scanner          # ✅ 이미 완성

fp = Scanner().  _fp(code)              # 지문 생성
hits = index.search(fp)                 # ← 계약 지점
verdict = judge.verify(hits, code)      # WP-C
report.to_pr_comment(verdict)           # WP-B
```

> **이 `Hit` 하나만 합의하면**, WP-A 담당자는 인덱스에만 집중하고
> WP-B/C 담당자는 **가짜 Index(mock)** 로 먼저 개발할 수 있다. → **완전 병렬**

## 4. 역할 분담 ✅ *(확정)*

| | **이우진** (팀장 · `woojin`) | **김민수** (`minsoo`) |
|---|---|---|
| **작업 패키지** | **A** 인덱스 · **C** 정확도 · **E** 벤치마크 | **D** 다언어 · **B** PR게이트 · **F** 배포 |
| **비중** | **약 70~75%** | 약 25~30% |
| **담당 파일** | `core/` `index/` `explain/` `benchmarks/` | `languages/` `adapters/` `report/` `cli.py`<br/>`.github/action/` `docs/` `README.md` |
| **성격** | 알고리즘 · 품질 · 평가 (**최난도**) | 확장 · 통합 · 배포 (**스펙 명확**) |

### 왜 이렇게 나눴나
- **팀장이 프로젝트의 두 축(인덱스 + 정확도)을 쥔다.** 심사에서 기술 난도를 방어하는 부분이 전부 여기 있다.
- **김민수는 막히지 않는 일을 맡는다.** 스펙이 명확하고 테스트로 정답을 즉시 확인할 수 있다.
- **WP-B(PR 게이트)는 데모의 주인공이다.** "PR을 열었더니 경고가 뜬다" — 시연영상에서 가장 화려한 장면.

### 김민수 태스크 순서 (난이도 사다리)
| ID | 태스크 | 난이도 |
|---|---|:--:|
| T-01 | Java 언어팩 | 하 |
| T-02 | JS/TS 언어팩 | 하 |
| T-03 | `provenire scan` CLI (+ mock Index) | 중 |
| T-04 | git diff 어댑터 | 중 |
| T-05 | **GitHub Action + PR 인라인 코멘트** ⭐ | 중 |
| T-06 | PyPI 배포 · README | 하 |

> 작업 지시서는 [`docs/tasks/`](tasks/) 에 있다. 팀장이 배정하고 `git pull` 로 받는다.

### ⚠️ 팀장의 선행 작업 (김민수를 막지 않기 위해 **먼저** 한다)
1. **`src/provenire/languages/` 구조 분리** → T-01을 열어준다
2. **`Index` 인터페이스 + mock 제공** → T-03을 열어준다 (인덱스 완성을 기다리지 않게)
