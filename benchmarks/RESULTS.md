# 검증 결과 — 이름을 바꿔도 잡아내는가?

> 실행: `python benchmarks/poc_winnowing.py` (누구나 재현 가능)
> 측정일: 2026-07-12

## 검증 명제

> **"AI가 GPL 코드를 변수명만 바꿔서 재현했을 때, 우리는 잡아내는가?"**

이것이 Provenire의 존재 이유다.
GitHub Copilot의 `public code filter`는 **문자열이 거의 같을 때만** 걸러내므로,
**변수명만 바꾸면 그대로 통과한다.**

## 실험 설계

| 항목 | 내용 |
|---|---|
| **원본** | `qutebrowser/utils/utils.py :: elide_filename()` — 실제 **GPL-3.0-or-later** 코드 |
| **엔진 A `raw`** | 주석·공백만 제거 후 비교 (= Copilot 필터 수준) |
| **엔진 B `tokens`** | **식별자를 `ID`로 익명화** 후 비교 (= Provenire) |
| **지표** | containment = `|공유 지문| / |의심 코드 지문|` |
| **파라미터** | **k=15** (k-gram), w=12 (winnowing 윈도우) — k는 E-1 측정으로 정했다(아래) |
| **임계값** | 30% 이상이면 "표절 의심" |

## 결과

| 변형 시나리오 | raw | **tokens** | 판정 |
|---|:--:|:--:|---|
| 1) 그대로 복사 | 100.0% | **100.0%** | 둘 다 탐지 |
| 2) 주석·독스트링 삭제 | 100.0% | **87.5%** | 둘 다 탐지 |
| 3) **변수·함수명 전부 변경** | **8.8%** | **100.0%** | ⭐ **tokens만 탐지** |
| 4) 이름변경 + 주석삭제 + 재포맷 | **8.8%** | **87.5%** | ⭐ **tokens만 탐지** |
| 5) 무관한 코드 (음성 대조군) | 0.0% | **0.0%** | 오탐 없음 |

### k-gram 길이별 민감도 (시나리오 3)

| k | raw | **tokens** | 음성 대조군 오탐 |
|:--:|:--:|:--:|:--:|
| 10 | 17.6% | **100.0%** | 0.0% |
| **15** | 8.8% | **100.0%** | 0.0% |
| 20 | 2.6% | **100.0%** | 0.0% |
| 25 | 0.0% | **100.0%** | 0.0% |
| 30 | 0.0% | **100.0%** | 0.0% |

**k를 어떻게 잡아도 `tokens`는 100% 탐지 / 0% 오탐.**
반면 `raw`는 k가 커질수록 **0%로 붕괴**한다.

> ⚠️ 이 표만 보면 "k는 아무래도 좋다"고 오해하기 쉽다. **아니다.**
> 이 시나리오(이름만 변경)에서만 k가 무관할 뿐, **실제 LLM 재생성에서는 k가 결정적**이다. → E-1 참조.

## 결론

> **변수명을 바꾸면 raw 방식(= Copilot 필터 수준)은 사실상 실패한다(8.8%, 임계값 30% 미만).
> 토큰 정규화 방식은 100% 탐지하면서 오탐은 0%다.**

식별자 익명화가 **핵심 해자(moat)** 임이 코드로 입증되었다.

## E-1 — 실제 LLM 재생성 스모크 테스트 (2026-07-16)

위 실험의 변형은 **스크립트로 만든 것**이라 현실성이 부족했다.
그래서 **LLM(Claude)에게 실제 카피레프트 함수를 보여주고 다시 쓰게 해서** 재측정했다.

### 설계

| 항목 | 내용 |
|---|---|
| **인덱스** | 실전 코퍼스 **475개 함수 청크 / 9개 카피레프트 프로젝트** |
| **대상** | Radicale·gramps·buildbot 의 로직 있는 함수 3개 (GPL-2.0/3.0) |
| **재생성 강도** | **L1** 이름만 · **L2** 이름+표현/구조 재배치(*전형적 AI 재생성*) · **L3** 다른 구조로 재구현 |
| **오탐 대조군** | 자체 작성 코드 5개 |

> 재생성 코드는 원본의 파생물일 수 있어 **저장소에 커밋하지 않는다**(§4). 수치와 방법론만 남긴다.

### 결과 — k값에 따른 탐지율

| k | L1 이름만 | **L2 표현변경** | L3 재구현 | 오탐(자체작성) |
|:--:|:--:|:--:|:--:|:--:|
| 25 | 3/3 | **0/3** | 0/3 | 0/5 |
| 20 | 3/3 | **0/3** | 0/3 | 0/5 |
| **15** | 3/3 | **2/3** | 0/3 | **0/5** |
| 12 | 3/3 | 1/3 | 0/3 | 0/5 |
| 10 | 3/3 | 2/3 | 1/3 | **1/5** ← 오탐 발생 |

### 발견

1. **k=25(기존 기본값)는 L2를 하나도 못 잡았다(0/3).** 이름과 함께 표현·구조가 바뀌면
   k-gram 해시가 전부 달라지기 때문이다. `if not A: if B:` → `if not A and B:` 같은 재배치에 취약하다.
2. **k=25는 raw와의 대비를 극대화하는 값이었지, 실전 탐지력 기준이 아니었다.**
   위 민감도 표에서 raw가 k=25일 때 0%로 붕괴하는 게 극적으로 보였을 뿐이다.
3. **k=15가 스윗스팟** — L2를 2/3 잡으면서 오탐은 0/5 유지. → **기본값을 15로 변경**했다.
4. k=10까지 낮추면 L3도 일부 잡지만 오탐이 나타난다(1/5). 과도하다.

### 한계 (정직하게)

- **샘플이 작다** (L2 3개 · 오탐 5개). k=12가 k=15보다 낮은 건 통계 노이즈로 보인다.
  정식 데이터셋(E-2/E-3)으로 재확인이 필요하다.
- **L3(다른 구조로 재구현)는 여전히 못 잡는다.** 다만 이는 사실상 독립 구현에 가까워
  "표절"로 볼지 자체가 논쟁적이다.
- 남은 대응책: **AST 구조 해시**. (호출 식별자 보존은 아래에서 측정해 **기각**했다)

### 개별 유사도 (k=15 기준, 참고)

| 대상 | L1 이름만 | L2 표현변경 | L3 재구현 |
|---|:--:|:--:|:--:|
| Radicale / `ssl_get_protocols` | 100% ✅ | 8.3% ❌ | 0% ❌ |
| gramps / `convert_21` | 100% ✅ | 30.0% ✅ | 0% ❌ (top 29%) |
| buildbot / `is_message_needed_by_results` | 100% ✅ | 35.7% ✅ | 17.8% ❌ |

> gramps L3 는 임계값 30% 에 **1%p 차이로** 못 미쳤다. 경계에 걸친 사례가 있다는 뜻이다.

## 작은 diff 오탐 (2026-07-20) — dogfooding 이 잡아낸 실전 문제

### 어떻게 발견했나

GitHub Action(T-05)을 **우리 저장소에 붙여둔 덕분**에 드러났다.
민수의 인코딩 수정 PR(#28)에 **우리 게이트가 오탐 10건**을 달았다:

```
[!] 표절 의심 10건 (임계값 30%)
  50.0%  src/provenire/adapters/git.py
     ↳ qutebrowser/utils.py :: yaml_dump  [GPL-3.0-or-later]
  50.0%  ... :: _tabs, window, undo, delete_tab
```

민수가 직접 쓴 코드이고 qutebrowser 와 아무 관계가 없다. **전부 오탐이다.**

### 원인 — containment 는 지문이 적을 때 무의미하게 부풀려진다

```
PR이 추가한 코드: 10줄 → 정규화 59자 → 지문 4개
겹친 지문 2개 → 2/4 = 50%  →  임계값 30% 초과
```

`scan --diff` 는 **PR에서 추가된 줄만** 검사하므로 작은 PR일수록 지문이 몇 개 안 나온다.
지문 1개 중 1개가 겹치면 **100%** 가 되기도 한다.

**우리 저장소 커밋 25개를 스캔한 결과 13개 조각 중 3개(23%)가 오탐이었다.**

### 대책 탐색 — 실측

정탐(카피레프트 함수 변형 108건) vs 오탐후보(우리 저장소 diff 13건)로 스윕했다.

| 최소 공유 지문 | 정탐 Recall | 오탐 |
|:--:|:--:|:--:|
| 1 (기존) | 98.1% | **3건** |
| **3** | **92.6%** | **0건** ← 채택 |
| 5 | 88.9% | 0건 |

"최소 **지문 수**" 조건(코드가 짧으면 판정 보류)도 함께 시험했으나 **Recall 만 크게 깎아** 기각했다
(하한 15면 Recall 53.7%). **최소 공유 지문 수**가 정확히 문제를 겨냥한다.

### 채택과 결과

`index/__init__.py` 의 `MIN_SHARED = 3` — 3개 미만으로 겹치는 후보는 검색 단계에서 제외한다.

| | 이전 | 이후 |
|---|:--:|:--:|
| 실전 오탐(우리 저장소) | 3건 (23%) | **0건** |
| Precision | 100% | 100% |
| Recall | 98.1% | **92.6%** |
| F1 | 99.1% | **96.2%** |

### 트레이드오프 (정직하게)

**정탐 8건을 잃었다.** 놓친 것은 `OctoPrint::analysis_backlog`(본문이 빈 함수),
`OctoPrint::get_size`(지문 3개 미만의 짧은 함수) — **원래 판정이 불안정한 조각들**이다.

그래도 손실은 손실이다. 이 선택의 근거는 하나다:
**PR 게이트에서 오탐은 사용자가 도구를 꺼버리게 만든다.** 놓친 표절은 다음 PR에서 다시 볼
기회가 있지만, 신뢰를 잃으면 그 기회 자체가 사라진다.

> 💡 **이 문제는 dogfooding 없이는 발견할 수 없었다.** 단위 테스트도, `evaluate.py` 도
> "작은 diff" 라는 실전 조건을 재현하지 못했다. 심사 시연 중에 터졌을 문제다.

## 큰 파일 미탐 (2026-07-20) — 코드 리뷰가 잡아낸 사각지대

### 증상 — GPL 파일 통째 복사가 통과된다

```
Radicale/pathutils.py (인덱스에 있는 GPL-3.0 원본) 을 그대로 복사해 scan
  → [OK] 카피레프트 유사 코드를 찾지 못했습니다.   exit 0   ← 통과!
```

가장 명백하고 흔한 표절("AI 가 GPL 파일을 통째로 뱉음")을 놓쳤다.

### 원인 — 인덱스와 쿼리의 입도 불일치

인덱스는 원본을 **함수 단위**로 저장하는데, `scan` 은 의심 코드를 **파일 통째**로 지문 떴다.
containment = `공유/의심전체` 라 분모가 폭발한다.

```
파일 전체 지문 543개 / 최고 히트 Radicale::acquire 76개 공유 → 14.0% (임계값 30% 미만)
같은 파일을 함수 단위로 쪼개 조회하면 → 19/19 전부 100% 탐지
```

역설적이다. `matcher.py` 는 *"작은 함수를 큰 GPL 파일에서 베낀 경우 containment 가
잡아낸다"* 고 하는데, **정반대(큰 파일이 의심 코드)에서 무너졌다.**

### 왜 안 드러났나 — 검증 하네스가 실사용 경로를 우회했다

- `evaluate.py` 의 `detected()` 가 `index.search()` 를 **직접** 호출 → 늘 함수 청크 단위 조회
- `test_cli.py` 는 **작은 단일 함수 파일**만 사용
- `MIN_SHARED=3` 튜닝도 "작은 diff" 만 봤지 "큰 파일" 은 안 봄

> **작은 조각 오탐은 잡았는데, 큰 조각 미탐은 아무도 안 봤다.**

### 수정 — 의심 코드도 청킹해 청크별 판정

`index/chunker.py` 신설: Python 은 함수 단위(ast), 그 외/파싱 실패는 **슬라이딩 윈도우**.
`scan_code()` 가 이걸 써서 파일을 통째로 지문 뜨지 않는다. `--diff` 의 불완전 조각도
슬라이딩 윈도우로 걸린다.

| | 이전 | 이후 |
|---|:--:|:--:|
| GPL 파일 통째 scan | **미탐 (exit 0)** | **탐지 (exit 1)** |
| 우리 저장소 diff 오탐 | 0 | **0 유지** |
| evaluate F1 | 96.2% | **96.2% 유지** |

회귀 테스트 `test_scan_detects_whole_file_copy` 로 고정했다.
`evaluate.detected()` 도 `scan_code()` 경유로 바꿔 하네스가 실사용 경로를 타게 했다.

### 후속 — 슬라이딩 윈도우가 오탐을 재발시켰다 (같은 날, dogfooding 이 또 잡음)

§2 수정 직후, **우리 Action 이 PR #36 자신에게 오탐 코멘트를 달았다.** cli.py 의 짧은
diff 조각(슬라이딩 윈도우, 지문 4개)에서 흔한 `for-if` 구조 3개가 ansible·weblate 함수와
우연히 겹쳐 75%(공유 3)로 잡혔다 — `MIN_SHARED=3` 의 경계를 딱 넘긴 것이다.

실측으로 대책을 비교했다(오탐: 우리 저장소 diff 22청크 / 정탐: 변형함수 108 + GPL파일통째 4):

| 대책 | 오탐 | 변형함수 정탐 | 파일통째 |
|---|:--:|:--:|:--:|
| `MIN_SHARED=3` (직전) | **1** | 100/108 | 4/4 |
| **`MIN_SHARED=4`** | **0** | 98/108 | 4/4 ← 채택 |
| `MIN_FP=8` (최소 지문 수) | 0 | **90/108** | 4/4 |

`MIN_FP`(짧으면 판정 보류)는 정탐을 크게 깎아 다시 기각했다(E-1 후속과 같은 결론).
`MIN_SHARED=4` 는 정탐 2건만 내주고 오탐을 없앤다.

> **미탐을 고치니 오탐이 났고, 그 오탐을 dogfooding 이 즉시 잡았다.**
> (덤: 리포트가 비-tty 로 나갈 때 ANSI 색상 코드가 코멘트에서 깨지던 것도 함께 고쳤다 —
>  `sys.stdout.isatty()` 일 때만 색을 켠다.)

> 💡 **dogfooding 이 오탐을, 코드 리뷰가 미탐을 잡았다.**
> "우리 게이트의 사각지대를 스스로 찾아 고쳤다" — 실패를 지우지 않는 이 문서의 연장선이다.

## E-1 후속 실험 — 호출 식별자 보존 (2026-07-16) · **기각**

### 가설

AI 는 변수명은 잘 바꿔도 **라이브러리·API 이름은 잘 바꾸지 않는다**
(`convert_person(...)`, `Math.floor(...)`). 그런데 우리는 이 이름까지 전부 `ID` 로 지운다.
**호출 대상 이름을 보존하면 L2/L3 탐지가 오르지 않을까?**

### 방법

`normalize_tokens(..., keep_calls=True)` — 뒤에 `(` 가 오는 식별자는 익명화하지 않는다
(언어 무관 휴리스틱: 어느 언어든 호출은 `이름(` 형태).

### 결과 — **개선 없음**

| 설정 | L1 이름만 | L2 표현변경 | L3 재구현 | 오탐 | 정규화 길이 |
|---|:--:|:--:|:--:|:--:|:--:|
| `keep_calls=False` (현행) | 3/3 | 2/3 | 0/3 | 0/5 | 375 |
| `keep_calls=True` | 3/3 | 2/3 | 0/3 | 0/5 | 414 |

**탐지율이 전혀 변하지 않았고, 정규화 길이만 늘었다**(= 지문·저장 비용 증가).

### 왜 안 됐나 (구현 문제가 아니다 — 확인함)

정규화 결과를 직접 열어 확인했다:

```
원본 convert_21 :  convert_person 보존됨 ✓
L2 재생성       :  convert_person 보존됨 ✓  ← 그런데도 탐지율 변화 없음
L3 재생성       :  convert_person 보존 안 됨 ✗
```

1. **k-gram 은 이름만이 아니라 주변 토큰 시퀀스가 함께 같아야 매칭된다.**
   이름을 살려도 구조(`if/elif` → `dict` 등)가 바뀌면 15글자 창이 여전히 어긋난다.
2. **L3 는 호출이 아니라 참조였다.** `{"Person": convert_person, ...}` 처럼 dict 값으로 쓰이면
   뒤에 `(` 가 오지 않아 휴리스틱이 놓친다.
3. 참조까지 보존하려면 사실상 **이름 익명화를 포기**하는 셈이라, 우리의 핵심 해자와 정면으로 충돌한다.

### 결론

**채택하지 않는다.** 이득 없이 코드 복잡도와 지문 비용만 늘어난다.
L2/L3 대응은 이름 보존이 아니라 **표현이 달라도 같아지는 표현**(AST 구조 해시)으로 가야 한다.

> 이 실험을 기록하는 이유: **같은 가설로 다시 시간을 쓰지 않기 위해서다.**

## 한계 (정직하게)

이 실험은 **Go/No-Go 판단용 PoC**다. 다음이 남아 있다.

1. ~~**단일 함수·단일 언어(Python)**~~ → **해소**: 코퍼스를 9개 프로젝트 475청크로 확대(A-1).
   단 청킹은 아직 Python(ast)만 함수 단위이고, 다른 언어는 파일 단위다.
2. ~~**변형을 스크립트로 생성**~~ → **해소**: E-1에서 실제 LLM 재생성으로 재측정.
   **그 결과 k=25의 취약점이 드러나 기본값을 15로 바꿨다.** (위 E-1 절 참조)
3. **대규모 인덱스에서의 오탐** — 475청크에서는 오탐 0/5로 측정됐으나 **표본이 작다.**
   관용구 필터(WP-C)는 구현했지만 이 표본에선 오탐이 애초에 0이라 효과가 드러나지 않았다.
   → 정식 오탐 세트(E-2)로 재측정 필요.
4. **L3(구조 재구현)는 미탐지** — AST 구조 해시 / 호출 식별자 보존으로 대응 예정.
