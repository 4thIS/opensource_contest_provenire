# 검증 결과 — 이름을 바꿔도 잡아내는가?

> 실행: `python benchmarks/poc_winnowing.py` (누구나 재현 가능)
> 측정일: 2026-07-12

## 검증 명제

> **"AI가 GPL 코드를 변수명만 바꿔서 재현했을 때, 우리는 잡아내는가?"**

이것이 Provenire의 존재 이유다.
GitHub Copilot의 `public code filter`는 **문자열이 거의 같을 때만** 걸러내므로,
**변수명만 바꾸면 그대로 통과한다.**

## 실험 설계

| 항목 | 내용 |
|---|---|
| **원본** | `qutebrowser/utils/utils.py :: elide_filename()` — 실제 **GPL-3.0-or-later** 코드 |
| **엔진 A `raw`** | 주석·공백만 제거 후 비교 (= Copilot 필터 수준) |
| **엔진 B `tokens`** | **식별자를 `ID`로 익명화** 후 비교 (= Provenire) |
| **지표** | containment = `|공유 지문| / |의심 코드 지문|` |
| **파라미터** | **k=15** (k-gram), w=12 (winnowing 윈도우) — k는 E-1 측정으로 정했다(아래) |
| **임계값** | 30% 이상이면 "표절 의심" |

## 결과

| 변형 시나리오 | raw | **tokens** | 판정 |
|---|:--:|:--:|---|
| 1) 그대로 복사 | 100.0% | **100.0%** | 둘 다 탐지 |
| 2) 주석·독스트링 삭제 | 100.0% | **87.5%** | 둘 다 탐지 |
| 3) **변수·함수명 전부 변경** | **8.8%** | **100.0%** | ⭐ **tokens만 탐지** |
| 4) 이름변경 + 주석삭제 + 재포맷 | **8.8%** | **87.5%** | ⭐ **tokens만 탐지** |
| 5) 무관한 코드 (음성 대조군) | 0.0% | **0.0%** | 오탐 없음 |

### k-gram 길이별 민감도 (시나리오 3)

| k | raw | **tokens** | 음성 대조군 오탐 |
|:--:|:--:|:--:|:--:|
| 10 | 17.6% | **100.0%** | 0.0% |
| **15** | 8.8% | **100.0%** | 0.0% |
| 20 | 2.6% | **100.0%** | 0.0% |
| 25 | 0.0% | **100.0%** | 0.0% |
| 30 | 0.0% | **100.0%** | 0.0% |

**k를 어떻게 잡아도 `tokens`는 100% 탐지 / 0% 오탐.**
반면 `raw`는 k가 커질수록 **0%로 붕괴**한다.

> ⚠️ 이 표만 보면 "k는 아무래도 좋다"고 오해하기 쉽다. **아니다.**
> 이 시나리오(이름만 변경)에서만 k가 무관할 뿐, **실제 LLM 재생성에서는 k가 결정적**이다. → E-1 참조.

## 결론

> **변수명을 바꾸면 raw 방식(= Copilot 필터 수준)은 사실상 실패한다(8.8%, 임계값 30% 미만).
> 토큰 정규화 방식은 100% 탐지하면서 오탐은 0%다.**

식별자 익명화가 **핵심 해자(moat)** 임이 코드로 입증되었다.

## E-1 — 실제 LLM 재생성 스모크 테스트 (2026-07-16)

위 실험의 변형은 **스크립트로 만든 것**이라 현실성이 부족했다.
그래서 **LLM(Claude)에게 실제 카피레프트 함수를 보여주고 다시 쓰게 해서** 재측정했다.

### 설계

| 항목 | 내용 |
|---|---|
| **인덱스** | 실전 코퍼스 **475개 함수 청크 / 9개 카피레프트 프로젝트** |
| **대상** | Radicale·gramps·buildbot 의 로직 있는 함수 3개 (GPL-2.0/3.0) |
| **재생성 강도** | **L1** 이름만 · **L2** 이름+표현/구조 재배치(*전형적 AI 재생성*) · **L3** 다른 구조로 재구현 |
| **오탐 대조군** | 자체 작성 코드 5개 |

> 재생성 코드는 원본의 파생물일 수 있어 **저장소에 커밋하지 않는다**(§4). 수치와 방법론만 남긴다.

### 결과 — k값에 따른 탐지율

| k | L1 이름만 | **L2 표현변경** | L3 재구현 | 오탐(자체작성) |
|:--:|:--:|:--:|:--:|:--:|
| 25 | 3/3 | **0/3** | 0/3 | 0/5 |
| 20 | 3/3 | **0/3** | 0/3 | 0/5 |
| **15** | 3/3 | **2/3** | 0/3 | **0/5** |
| 12 | 3/3 | 1/3 | 0/3 | 0/5 |
| 10 | 3/3 | 2/3 | 1/3 | **1/5** ← 오탐 발생 |

### 발견

1. **k=25(기존 기본값)는 L2를 하나도 못 잡았다(0/3).** 이름과 함께 표현·구조가 바뀌면
   k-gram 해시가 전부 달라지기 때문이다. `if not A: if B:` → `if not A and B:` 같은 재배치에 취약하다.
2. **k=25는 raw와의 대비를 극대화하는 값이었지, 실전 탐지력 기준이 아니었다.**
   위 민감도 표에서 raw가 k=25일 때 0%로 붕괴하는 게 극적으로 보였을 뿐이다.
3. **k=15가 스윗스팟** — L2를 2/3 잡으면서 오탐은 0/5 유지. → **기본값을 15로 변경**했다.
4. k=10까지 낮추면 L3도 일부 잡지만 오탐이 나타난다(1/5). 과도하다.

### 한계 (정직하게)

- **샘플이 작다** (L2 3개 · 오탐 5개). k=12가 k=15보다 낮은 건 통계 노이즈로 보인다.
  정식 데이터셋(E-2/E-3)으로 재확인이 필요하다.
- **L3(다른 구조로 재구현)는 여전히 못 잡는다.** 다만 이는 사실상 독립 구현에 가까워
  "표절"로 볼지 자체가 논쟁적이다.
- 남은 대응책: **호출 식별자(라이브러리 API명) 보존** — 재생성물이 `convert_person` 등
  동일 API를 호출하는데 현재는 전부 `ID`로 지워 신호를 버린다. 그리고 **AST 구조 해시**.

## 한계 (정직하게)

이 실험은 **Go/No-Go 판단용 PoC**다. 다음이 남아 있다.

1. ~~**단일 함수·단일 언어(Python)**~~ → **해소**: 코퍼스를 9개 프로젝트 475청크로 확대(A-1).
   단 청킹은 아직 Python(ast)만 함수 단위이고, 다른 언어는 파일 단위다.
2. ~~**변형을 스크립트로 생성**~~ → **해소**: E-1에서 실제 LLM 재생성으로 재측정.
   **그 결과 k=25의 취약점이 드러나 기본값을 15로 바꿨다.** (위 E-1 절 참조)
3. **대규모 인덱스에서의 오탐** — 475청크에서는 오탐 0/5로 측정됐으나 **표본이 작다.**
   관용구 필터(WP-C)는 구현했지만 이 표본에선 오탐이 애초에 0이라 효과가 드러나지 않았다.
   → 정식 오탐 세트(E-2)로 재측정 필요.
4. **L3(구조 재구현)는 미탐지** — AST 구조 해시 / 호출 식별자 보존으로 대응 예정.
